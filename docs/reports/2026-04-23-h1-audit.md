# H1 Styling And Structure Audit

Date: 2026-04-23

## Summary

The site already has a mostly shared H1 system.

- The base HTML heading rules live in `src/styles.css`.
- The actual page-opening H1 pattern is overwhelmingly driven by the shared `.hero-display` class in `src/styles.css`.
- Most live pages use `.hero-display` on the real page `<h1>`.
- The main inconsistency is not wholesale visual drift. It is that pages make small page-scoped adjustments in different ways:
  - title measure via page-specific `max-width`
  - margin-bottom tweaks on the title
  - different surrounding hero structures
  - different eyebrow or badge treatments above the H1

So the current state is: shared foundation, but with a growing layer of page-specific overrides and hero markup variations.

## What Is Shared Today

### 1. Base heading typography

`src/styles.css` defines the baseline site typography:

- `h1, h2, h3, .brand__name` share the serif family and weight
- `h1` sets the default large heading scale, line-height, and max-width

This is the semantic HTML baseline, but it is not the main production H1 pattern used on live page heroes.

### 2. Shared hero-title system

The real shared H1 system is:

- `.hero-section`
- `.hero-top`
- `.hero-display`
- related support primitives like `.hero-copy-panel`, `.hero-intro`, `.hero-badge`, `.hero-support-tagline`

`src/styles.css` makes `.hero-display` the canonical display title:

- large serif scale
- lighter weight
- balanced wrapping
- shared color
- default bottom margin

This is also documented as the canonical production hero heading in `src/pages/design-system/DS_Heroes.tsx`.

### 3. Shared dev and design-system H1 handling

`src/components/DevPageHero.tsx` renders:

- `.hero-badge`
- `<h1 className="hero-display">`
- `.hero-copy-panel`

That means the dev pages are already normalized onto one H1 implementation.

## What Is Customised

### 1. Title measure

Several pages customize only the H1 width:

- `src/styles-home.css`
- `src/styles-working-with-joel.css`
- `src/styles-enm-polyamory.css`
- `src/styles-kink-bdsm.css`

This is the most common override. It suggests the shared system is correct, but measure is still being tuned ad hoc per page.

### 2. Title spacing

A few pages change the default `.hero-display` bottom rhythm:

- `src/styles-enquire.css` uses `.enquire-page .hero-display { margin-bottom: 24px; }`
- `src/styles-enm-polyamory.css` sets `margin-bottom: 0`
- `src/styles-kink-bdsm.css` sets `margin: 0`
- `src/styles-lgbtqia.css` sets `margin-bottom: 0`
- `src/styles-inclusive-practice.css` sets `margin: 0` on the title wrapper

These are all small changes, but they are handled in different ways.

### 3. Eyebrow or badge treatment above the H1

There is not one fully normalized pre-title pattern.

- Shared pattern: `.hero-badge`
- Custom page pattern: `.kink-page__hero-eyebrow`
- No eyebrow at all on some pages
- Breadcrumbs above the H1 on some inclusion pages

This creates structural variation even when the H1 itself is shared.

### 4. Hero layout around the H1

The H1 usually lives inside the shared hero shell, but the surrounding structure varies:

- Home and Enquire use `hero-top--supporting-media`
- InclusivePractice has a more bespoke masthead and caption structure
- Kink and ENM place supporting intro copy under the title in the left column
- Some pages put support copy in `.hero-copy-panel`
- Some use `.hero-intro`
- Some use custom paragraph classes

This is where most of the divergence actually lives.

## Key Findings

### 1. The site does already have one de facto H1 style

That style is `.hero-display`, not raw `h1`.

This is good news. The normalization problem is smaller than it first appears.

### 2. The main drift is in page-scoped title adjustments

Most title customizations are not redefining font family, weight, or scale. They are adjusting:

- width
- bottom spacing
- line breaking context

That means the problem is more about managing variants than replacing the system.

### 3. Structure is less normalized than typography

The H1 look is reasonably shared.

The hero composition around the H1 is less consistent:

- badge vs eyebrow vs breadcrumb + badge
- intro under title vs support panel on the right
- different custom wrappers around essentially similar title stacks

### 4. Raw `h1` styling is not the production source of truth

The base `h1` rule still matters, but it is not what most real pages depend on.

If future work treats raw `h1` and shared page H1 styling as the same thing, that will create confusion.

## Recommendations

### 1. Make `.hero-display` the explicit authoritative site H1 pattern for page openings

Document this clearly:

- raw `h1` = baseline semantic heading styling
- `.hero-display` = authoritative production page-opening H1 styling

That matches the real codebase already.

### 2. Replace page-by-page title tweaks with a small shared modifier system

Right now pages individually solve the same small problems.

Normalise these into shared modifiers or variables, for example:

- standard title rhythm
- flush title rhythm
- narrower title measure
- wider title measure

This would remove one-off rules like:

- `.enquire-page .hero-display`
- `.lgbtqia-page__hero-title`
- `.enm-page__hero-title`
- `.working-with-joel-page__hero-title`

for cases where the only real difference is width or margin.

### 3. Prefer shared hero title settings via variables over page-specific title classes

A good normalization direction would be:

- keep `.hero-display` as the one visual rule
- let pages set local variables such as title measure or margin
- reserve page-specific H1 classes for genuinely unique structures only

That would simplify maintenance and make hero-title changes safer across the site.

### 4. Define one canonical title-stack structure

Document a preferred order for the left column when a hero is text-led:

1. breadcrumb, if needed
2. badge or eyebrow
3. H1
4. intro or support copy
5. trust row or actions

Pages can still omit parts, but the order should stop drifting.

### 5. Normalize the pre-title treatment

Choose one shared pattern for the small label above the H1:

- preferably `.hero-badge` for hero metadata
- use breadcrumbs separately when needed

Avoid page-only replacements like `.kink-page__hero-eyebrow` unless there is a strong reason the shared class cannot work.

### 6. Reduce "same outcome, different wrapper" hero markup

Some pages are already close enough that they could converge around the same left-column recipe:

- `Home`
- `WorkingWithJoel`
- `Enquire`
- `EnmPolyamoryCounselling`
- `LgbtqiaCounselling`

`InclusivePractice` is the clearest candidate to remain intentionally bespoke.

`KinkBdsmCounselling` may also justify a stronger editorial variant, but its H1 could still stay inside a more standardized title stack.

### 7. Keep page-level hero customization focused on composition, not title styling

Page CSS should mostly answer:

- column widths
- media placement
- secondary content placement

It should not keep accumulating separate local opinions about H1 spacing and title measure if those can be shared.

## Practical Normalization Plan

### Near-term

- Treat `.hero-display` as the authoritative page-opening H1 style in the docs
- Audit every page-specific H1 class and classify it as either:
  - measure only
  - spacing only
  - genuinely bespoke
- Collapse simple measure and spacing overrides into shared modifiers or variables

### Medium-term

- Define one documented "hero title stack" pattern in the design system
- Normalize pre-title labels onto shared classes
- Reduce page-only H1 classes where they are only setting width or bottom margin

### Keep As Intentional Exceptions

- `InclusivePractice` can remain more bespoke because it is effectively a hub page with a more editorial opening
- design-system demos can continue using `.hero-display` on `h2` where the component is being demonstrated rather than used as the document H1

## Bottom Line

The site is already closer to a normalized H1 system than it may feel.

There is one clear shared visual H1 pattern across the site: `.hero-display`.

The maintainability issue is mostly that small layout and rhythm variations are being expressed as page-specific title classes and local hero wrappers instead of a tighter shared variant system. Normalizing those variants should give a large maintainability win without needing a full redesign of page openings.
